# A Brief Introduction to Seata and Its Four Transaction Modes

## 1. What is Seata?

**Seata** (Simple Extensible Autonomous Transaction Architecture) is an open-source distributed transaction solution originally developed by Alibaba and now an Apache Incubator project. It is designed to provide high-performance and easy-to-use distributed transaction services in microservice architectures.

Before its open-source release, Seata (then an internal Alibaba product) played a critical role during the "Double 11" shopping festivals, ensuring data consistency across numerous microservices. In January 2019, Seata was officially open-sourced to create a more complete technological ecosystem.

> Seata（Simple Extensible Autonomous Transaction Architecture）是一款开源的分布式事务解决方案，最初由阿里巴巴开发，现为Apache孵化器项目。它致力于在微服务架构下提供高性能和简单易用的分布式事务服务。

### 1.1 Core Architecture: The Three Roles

Seata's architecture consists of three fundamental components that work together to manage distributed transactions:

- **TC (Transaction Coordinator) - 事务协调者**: The central server (seata-server) that maintains the states of global and branch transactions and drives global commit or rollback decisions.
- **TM (Transaction Manager) - 事务管理器**: The client component embedded in the application that defines the scope of a global transaction, begins it, and ultimately requests the TC to commit or rollback.
- **RM (Resource Manager) - 资源管理器**: The client component that manages branch transaction resources, registers branch transactions with the TC, and reports their status.

When a distributed transaction begins, the TM requests the TC to generate a globally unique **XID**. This XID propagates through the entire microservice invocation chain, associating all participating sub-transactions (managed by RMs) under the same global transaction.

## 2. The Four Transaction Modes

Seata offers four distinct distributed transaction modes to address different business requirements, ranging from strong consistency to high availability scenarios.

### 2.1 AT Mode (Auto Transaction)

AT mode is Seata's default and most popular solution, providing **non-intrusive** automatic compensation for distributed transactions.

**How it works:**
- **Phase 1**: The RM executes business SQL and commits the local transaction immediately. Before execution, Seata automatically generates a "before image" of the data; after execution, it generates an "after image." Both images are stored in an `undo_log` table.
- **Phase 2 (Commit)**: If the global transaction succeeds, the TC instructs all RMs to asynchronously delete their corresponding undo logs.
- **Phase 2 (Rollback)**: If any branch fails, the TC instructs all RMs to roll back. Each RM uses its saved undo log to restore the data to its original state.

**Supported Databases**: MySQL, Oracle, PostgreSQL, TiDB, MariaDB, Dameng, SQLServer.

**Advantages:**
- **Zero code invasion**: You only need to write your business SQL.
- **High performance**: Phase 1 commits immediately, releasing database locks quickly.

**Limitations:**
- Provides **eventual consistency**, not strong consistency.
- Requires undo log management, which adds slight overhead.

> AT模式是Seata默认的、无侵入的分布式事务解决方案。它通过记录数据快照（undo log）实现自动补偿，一阶段快速提交释放资源，二阶段根据全局结果进行提交或回滚。

### 2.2 TCC Mode (Try-Confirm-Cancel)

TCC mode requires the business to define three explicit phases, offering more flexibility at the cost of business code intrusion.

**The Three Phases:**
- **Try**: Reserve resources and check business conditions. (e.g., check and freeze inventory)
- **Confirm**: Commit the business operation if all Try phases succeed. Uses the reserved resources from the Try phase. (e.g., deduct the frozen inventory)
- **Cancel**: Roll back and release the reserved resources if any Try phase fails. (e.g., release the frozen inventory)

**Advantages:**
- High performance: Phase 1 (Try) completes and commits quickly.
- No global lock or undo log overhead.
- Flexible: Suitable for scenarios where business logic needs explicit control.

**Limitations:**
- **Code intrusion**: You must implement Try, Confirm, and Cancel logic manually.
- Requires careful handling of idempotence and cancellation scenarios.

### 2.3 XA Mode

XA mode implements the **XA protocol** standard, providing **strong consistency** through a traditional two-phase commit mechanism.

**How it works:**
- **Phase 1**: The RM executes the branch transaction but **does not commit** it. Instead, it holds the database locks and reports its status to the TC.
- **Phase 2**: The TC decides based on all branch statuses. If all succeed, it instructs all RMs to commit; otherwise, all roll back.

**Supported Databases**: MySQL, Oracle, PostgreSQL, MariaDB.

**Advantages:**
- **Strong consistency**: Satisfies ACID principles.
- **Zero code invasion**: Leverages native database XA support.
- Simple to implement.

**Limitations:**
- **Lower performance**: Database resources remain locked throughout the entire transaction, waiting for Phase 2.
- Dependent on relational databases that support XA.

> XA模式基于标准的XA协议，采用两阶段提交实现强一致性事务。它无代码侵入，但性能相对较低，因为数据库资源需要在整个事务期间保持锁定。

### 2.4 SAGA Mode

SAGA mode is designed for **long-running transactions** and complex business processes where transactions might take seconds, minutes, or even longer to complete.

**How it works:**
SAGA breaks a global transaction into a sequence of local sub-transactions. Each sub-transaction has a corresponding **compensation transaction** (rollback operation). If a step fails, SAGA executes the compensation transactions of all previously completed steps in reverse order.

**Advantages:**
- Suitable for long-running and complex business workflows.
- Participants can execute asynchronously, improving system availability.
- No resource locking between steps.

**Limitations:**
- **Code intrusion**: Requires designing and implementing compensation logic.
- Potential risk of data inconsistency between steps (though compensable).
- Compensation logic can be complex to design correctly.

## 3. Quick Comparison Summary

| Mode | Consistency | Code Invasion | Performance | Typical Use Case |
|:---|:---|:---|:---|:---|
| **AT** | Eventual | None | High | CRUD operations, general microservices |
| **TCC** | Eventual | High | Very High | Business with explicit resource reservation |
| **XA** | Strong | None | Lower | Financial systems requiring ACID |
| **SAGA** | Eventual | High | Medium | Long-running processes, complex workflows |

## 4. When to Choose Which?

- **Choose AT mode** if you want a non-intrusive solution with good performance and can accept eventual consistency. This covers most general business scenarios.
- **Choose TCC mode** when you need high performance and can invest in writing Try/Confirm/Cancel logic, especially for scenarios like account freezing or inventory reservation.
- **Choose XA mode** when **strong consistency** is non-negotiable (e.g., financial transactions) and you can tolerate lower throughput.
- **Choose SAGA mode** for long-running transactions or when business processes involve multiple heterogeneous systems that cannot be modified easily.

> 总结：Seata通过四种事务模式覆盖了从强一致到最终一致、从无侵入到高可控的各种分布式事务场景。AT模式适合大多数业务，TCC适合高性能场景，XA适合强一致场景，SAGA适合长事务场景。根据具体业务需求和数据一致性要求选择合适的模式是关键。

## 5. Ecosystem and Integration

Seata integrates seamlessly with popular microservice frameworks including **Spring Cloud, Dubbo, gRPC, and Sofa-RPC**. It also works well with ORM frameworks like **MyBatis, MyBatis-Plus, JPA, and Hibernate** through DataSource proxying. The project offers native **GoLang** implementations and supports Kubernetes deployment.
